Every application eventually has to answer the same question: **how should we store this data?** A shopping cart, a chat history, a grant application - the features differ, but the underlying problem is the same. You need a structure that is accurate, efficient, and easy to change as requirements evolve.

That structure is a **data model**. It is the blueprint for how information is organized, related, and constrained inside a database. Get it wrong early and you pay for it in slow queries, duplicated records, and migrations that take weeks. Get it right and the rest of the stack - APIs, analytics, reporting - becomes much simpler.

This post walks through the core ideas: what data modeling is, the main types of models, how ER diagrams visualize relationships, what normalization solves, and how conceptual, logical, and physical design fit together.

## What is data modeling?

**Data modeling** is the process of defining what data your system needs, how entities relate to each other, and what rules govern that data - before you write a single `CREATE TABLE` statement.

A good model captures:

- **Entities** - the "things" you track (users, orders, products, sessions)
- **Attributes** - properties of those things (email, price, created_at)
- **Relationships** - how entities connect (a user *places* many orders; an order *contains* many line items)
- **Constraints** - rules the database enforces (unique emails, non-null foreign keys, valid status values)

You are not just drawing boxes. You are making decisions that affect query performance, data integrity, and how painful future changes will be.

### Why it matters

| Without a clear model | With a clear model |
|---|---|
| Duplicate data across tables | Single source of truth per fact |
| Orphaned records and inconsistent state | Foreign keys and constraints keep data honest |
| Queries that need 6 joins to answer a simple question | Relationships designed for common access patterns |
| Schema changes that break production | Layers of design let you iterate before committing |

Whether you are on Postgres, MySQL, or MongoDB, the thinking is the same. Relational databases make constraints explicit; document stores still need the same entity and relationship decisions - they just express them differently.

## Types of data models

Data models are often grouped by **level of abstraction** and by **underlying structure**. Both matter.

### By abstraction level

1. **Conceptual** - business view. What exists and how it relates, without technical detail.
2. **Logical** - technology-agnostic structure. Entities, attributes, keys, and cardinalities, but no storage engine specifics.
3. **Physical** - implementation. Table names, column types, indexes, partitions, and storage choices.

These three layers are the backbone of database design. More on them in a dedicated section below.

### By data structure

**Relational model** - data lives in tables with rows and columns. Relationships use primary and foreign keys. This is the default for most transactional apps (e-commerce, SaaS, banking).

**Dimensional model** - optimized for analytics. Fact tables hold measurable events (sales, clicks); dimension tables hold descriptive context (date, product, region). Common in data warehouses and BI.

**Document model** - nested JSON-like documents. Good when data is read and written as a unit (user profiles with preferences, CMS content). Relationships can be embedded or referenced by ID.

**Graph model** - nodes and edges. Fits social networks, recommendation engines, and anything where *who is connected to whom* is the primary query pattern.

**Key-value / wide-column** - simple or highly distributed lookups at massive scale. Think session stores, feature flags, or time-series telemetry.

Most production systems use more than one. A Postgres app might sync events to a warehouse built on a dimensional model, while a Redis layer caches hot key-value reads.

## Entity-Relationship (ER) diagrams

An **ER diagram** is the standard visual language for data models. It answers: *what are the nouns, what do we know about them, and how do they link?*

### Core notation

| Symbol | Meaning |
|---|---|
| Rectangle | **Entity** - a distinct object type (Customer, Order) |
| Oval / attribute list | **Attribute** - a property (email, total_amount) |
| Diamond | **Relationship** - a verb connecting entities (places, contains) |
| Lines + cardinality | **How many** on each side (one-to-many, many-to-many) |

### Cardinality

Cardinality describes how many instances on one side of a relationship map to the other:

- **One-to-one (1:1)** - a User has one Profile
- **One-to-many (1:N)** - a Customer places many Orders
- **Many-to-many (M:N)** - Students enroll in many Courses; each Course has many Students

Many-to-many relationships almost always need a **junction table** (also called an associative or bridge table) in a relational database:

```
┌──────────┐       ┌─────────────────┐       ┌──────────┐
│ Student  │───M:N─│ student_course  │─N:M───│ Course   │
│          │       │ student_id (FK) │       │          │
│ id (PK)  │       │ course_id (FK)  │       │ id (PK)  │
└──────────┘       └─────────────────┘       └──────────┘
```

### A small example

An online bookstore might look like this at the logical level:

```

│ Customer │───────────1:N───────────│  Order   │
│          │                         │          │
│ id       │                         │ id       │
│ email    │                         │ total    │

                                     │ OrderItem│

                                     └────┬─────┘

```

ER diagrams are most useful in the **conceptual** and **logical** phases - when you are aligning with stakeholders and validating that the model actually reflects the business. Once you move to physical design, the same relationships become foreign keys, indexes, and migration files.

## Normalization

**Normalization** is a set of rules for organizing columns and tables so that data is stored once, updates stay consistent, and anomalies do not creep in.

The problems normalization targets are called **update, insert, and delete anomalies**:

- **Update anomaly** - changing a customer's city requires updating rows in multiple places
- **Insert anomaly** - you cannot add a product until someone orders it
- **Delete anomaly** - deleting an order accidentally removes product information you still need

### Normal forms (the ones you actually use)

**First Normal Form (1NF)** - each column holds atomic (indivisible) values; no repeating groups.

Bad: `tags: "react,typescript,nextjs"` in one cell.

Better: a separate `post_tags` table, or a proper array type if your database supports it and your access patterns justify it.

**Second Normal Form (2NF)** - 1NF, and every non-key attribute depends on the *whole* primary key (relevant for composite keys).

If `order_item(order_id, product_id, product_name, quantity)` uses a composite key, `product_name` only depends on `product_id` - not the full key. Move `product_name` to `product`.

**Third Normal Form (3NF)** - 2NF, and no non-key attribute depends on another non-key attribute (no transitive dependencies).

If `employee(employee_id, department_id, department_name)` stores `department_name` alongside `department_id`, that name depends on `department_id`, not directly on `employee_id`. Split into `employee` and `department`.

**Boyce-Codd Normal Form (BCNF)** - a stricter 3NF. Every determinant is a candidate key. Useful when you have overlapping composite keys or subtle dependency edge cases.

In practice, most OLTP schemas aim for **3NF or BCNF**. Going further (4NF, 5NF) is rare outside specialized domains.

### When to denormalize

Normalization is not a religion. Read-heavy workloads - dashboards, feeds, search indexes - sometimes **denormalize** on purpose: duplicate a `product_name` on `order_item` to avoid a join, or maintain a `order_count` counter on `customer` for fast display.

The tradeoff is always the same: **faster reads vs. harder writes and risk of stale copies**. Denormalize when you have measured the need, not by default.

## Conceptual, logical, and physical design

Database design is usually done in three layers. Each layer answers a different question and has a different audience.

### Conceptual design

**Question:** What does the business care about?

**Audience:** Product owners, domain experts, engineers getting aligned.

**Output:** High-level ER diagrams, glossary of terms, relationship cardinalities. No column types, no indexes, no "should this be VARCHAR(255)?"

At this stage you might discover that "Order" means something different to finance and to fulfillment. Fixing that ambiguity here saves real money.

Example decisions:

- A `Subscription` belongs to one `Account`
- An `Invoice` can cover multiple `Subscription` periods
- `Refunds` are always tied to a single `Payment`

### Logical design

**Question:** What is the precise structure, independent of any one database engine?

**Audience:** Engineers, data architects.

**Output:** Fully specified entities, attributes, primary keys, foreign keys, unique constraints, and cardinalities. Still no `INTEGER` vs `BIGINT` debates - that is physical.

Example for the bookstore:

| Entity | Key | Notable attributes |
|---|---|---|
| Customer | `customer_id` | `email` (unique), `created_at` |
| Order | `order_id` | `customer_id` (FK), `status`, `placed_at` |
| OrderItem | (`order_id`, `product_id`) | `quantity`, `unit_price` |
| Product | `product_id` | `sku` (unique), `name`, `list_price` |

The logical model is what you validate against use cases: "Can we list all orders for a customer in the last 30 days?" "Can we prevent two active subscriptions on the same plan?"

### Physical design

**Question:** How does this actually run in production?

**Audience:** DBAs, backend engineers, DevOps.

**Output:** DDL scripts, index strategy, partitioning, replication, backup policy, and performance tuning.

```sql
CREATE TABLE orders (
  id          BIGSERIAL PRIMARY KEY,
  customer_id BIGINT NOT NULL REFERENCES customers(id),
  status      TEXT NOT NULL CHECK (status IN ('pending', 'paid', 'shipped', 'cancelled')),
  total_cents INTEGER NOT NULL CHECK (total_cents >= 0),
  placed_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_orders_customer_placed
  ON orders (customer_id, placed_at DESC);
```

Physical design is where abstractions meet reality:

- `BIGSERIAL` vs UUID for primary keys (insert locality vs. global uniqueness)
- Indexing foreign keys and common filter columns
- Choosing `TIMESTAMPTZ` over `TIMESTAMP` for anything user-facing
- Soft deletes (`deleted_at`) vs. hard deletes
- Read replicas and caching layers sitting on top of the schema

A common mistake is jumping straight to physical design - picking table names and types before the business relationships are settled. The three layers exist so you can change your mind cheaply at the top and commit only when the model is stable.

## Putting it together

A sensible workflow looks like this:

1. **Interview the domain** - nouns, verbs, rules, edge cases
2. **Draw a conceptual ER diagram** - align with stakeholders
3. **Refine to a logical model** - keys, constraints, junction tables for M:N
4. **Normalize to 3NF** - eliminate redundancy and anomalies
5. **Map to a physical schema** - types, indexes, migrations
6. **Denormalize selectively** - only where profiling proves it is worth it

Data modeling is not paperwork. It is the discipline of making your data tell the truth, stay consistent under concurrent writes, and remain understandable six months later when you have forgotten why `status` has four values instead of five.

If you are starting a new feature or greenfield service, spend an hour on paper (or a whiteboard tool) before you run `npx drizzle-kit generate` or write your first Prisma schema. The database is the part of your stack that is hardest to unwind - design it deliberately.
